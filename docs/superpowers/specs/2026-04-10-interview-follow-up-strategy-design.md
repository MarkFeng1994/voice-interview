# 模拟面试追问策略增强设计

**日期**: 2026-04-10

**范围**: 仅覆盖后端“面试流程与追问策略增强”的第一阶段，不包含前端展示、报告解释、RAG、agent。

## 1. 背景

当前模拟面试链路已经具备：

- 会话创建、答题、报告生成
- 简历定制题目
- `LangChain4j` 驱动的 LLM 话术生成
- 基础 `InterviewAnswerAnalyzer` 启发式判断

但现有追问决策仍然偏粗糙，主要问题是：

- 该追问时没有追问
- 不该追问时仍然继续深挖
- 追问次数控制粗糙，容易拖慢节奏
- 决策原因不够结构化，难以稳定测试和后续扩展

本设计的目标是提升后端追问决策准确性，同时保持主面试链路稳定可控。

## 2. 目标

- 让“是否追问”主要由后端规则稳定决定，而不是交给 LLM 自由发挥
- 将追问判定从单一启发式升级为“关键点 + 深度 + 正确性风险”的混合判断
- 在普通题、高价值题、开场题、收尾题之间形成不同的追问节奏
- 为后续报告解释、RAG、agent 扩展保留清晰挂点

## 3. 非目标

本阶段不做以下内容：

- 不修改移动端、管理端展示
- 不在报告页展示“为什么追问/跳题”
- 不引入向量检索或知识库
- 不引入 agent runtime
- 不让 LLM 主导“要不要追问”的最终决策

## 4. 总体方案

追问策略拆成三层：

1. **证据分析层**
   - 负责从题目、回答、期望点中提取结构化证据
   - 只回答“回答质量如何”，不直接决定流程分支

2. **流程决策层**
   - 负责结合分析结果、阶段、追问次数、题目价值，决定：
     - `FOLLOW_UP`
     - `NEXT_QUESTION`
     - `END_INTERVIEW`

3. **LLM 话术层**
   - 仅负责生成追问文案、过渡语、收尾语
   - 不再拥有最终追问控制权

这意味着：

- 是否追问：规则层决定
- 追问怎么说：LLM 生成
- 流程不会因 LLM 随机性而飘动

## 5. 核心判定模型

### 5.1 关键点完整度

输出三档：

- `HIGH`
- `MEDIUM`
- `LOW`

判定依据：

- `expectedPoints` 覆盖率
- 题目标题中可提炼的关键短语
- 题干中可提炼的关键短语
- 关键点别名/去后缀匹配

规则说明：

- 关键点基本覆盖，判 `HIGH`
- 只覆盖一部分，仍缺核心点，判 `MEDIUM`
- 大面积缺失或几乎未答到题上，判 `LOW`

### 5.2 回答深度

输出三档：

- `DEEP`
- `NORMAL`
- `SHALLOW`

第一阶段使用规则而非复杂语义评估，重点观察：

- 是否只有结论，没有过程
- 是否缺少“为什么 / 怎么做 / 如何权衡 / 例子 / 结果”
- 是否只有名词堆砌，没有落到经验或方法

推荐规则：

- 回答极短，且无过程/案例表达，判 `SHALLOW`
- 有过程或方法，但细节有限，判 `NORMAL`
- 有过程、取舍、案例、结果中的多类信息，判 `DEEP`

### 5.3 正确性风险

输出三档：

- `CONSISTENT`
- `SUSPECTED_CONTRADICTION`
- `CLEARLY_WRONG`

第一阶段只关注可稳定识别的风险：

- 前后自相矛盾
- 对题目主旨明显答偏
- 回答与题目目标严重不相关

不做强技术事实裁判，以免误判过高。

## 6. 决策矩阵

第一阶段采用“平衡型”策略。

### 6.1 普通题

- `LOW 完整度` -> 追问
- `MEDIUM 完整度 + SHALLOW` -> 追问
- `HIGH 完整度 + SHALLOW` -> 默认不追问，直接下一题
- `存在明显矛盾/错误风险` -> 优先追问澄清
- 已追问 1 次后仍低质量 -> 直接下一题

### 6.2 高价值题

高价值题允许更深挖，但仍有限制：

- `LOW/MEDIUM 完整度` -> 优先追问
- `HIGH 完整度 + SHALLOW` -> 允许追问一次补细节
- 追问次数上限 2 次
- 第二次仍无增量时直接跳题

### 6.3 开场题

- 开场阶段追问更保守
- 自我介绍类问题默认最多 0 到 1 次追问
- 只有明显未回答、完全跑题、或高度含混时才追问

### 6.4 收尾题

- 收尾阶段默认降低追问概率
- 仅在回答几乎无效或明显矛盾时才追问
- 否则优先保持节奏，进入下一题或结束

## 7. 追问方向

第一阶段只允许 3 种追问方向：

- `MISSING_KEY_POINT`
- `NEED_EXAMPLE_OR_DETAIL`
- `CLARIFY_CONTRADICTION`

好处：

- 后端逻辑简单明确
- 便于测试
- 后续报告展示时易于解释

## 8. 代码改动边界

### 8.1 保留现有边界

以下模块继续存在，但职责会收窄：

- `com.interview.module.ai.service.AiService`
- `com.interview.module.ai.service.langchain4j.LangChain4jAiService`
- `com.interview.module.ai.service.OpenAiCompatibleAiService`
- `com.interview.module.interview.controller.InterviewController`

### 8.2 扩展模块

#### `InterviewAnswerAnalyzer`

从当前单一启发式扩展成“证据分析器”，负责输出：

- 完整度
- 深度
- 正确性风险
- 缺失关键点
- 推荐追问方向
- 分析原因代码

#### `InterviewFlowPolicy`

扩展为流程策略配置中心，除时长规则外，还负责：

- 普通题追问上限
- 高价值题追问上限
- 开场/收尾阶段追问阈值
- 高价值题判定规则

#### `SimpleInterviewEngine`

只负责编排：

- 读取当前题目和会话状态
- 调用分析器
- 调用追问决策器
- 调用 `AiService` 生成追问或过渡文案
- 写入轮次记录

### 8.3 新增模块

建议新增：

- `FollowUpDecisionEngine`
- `AnswerEvidence`
- `FollowUpDecision`

职责边界：

- `AnswerEvidence`: 分析层输出对象
- `FollowUpDecision`: 决策层输出对象
- `FollowUpDecisionEngine`: 从证据到流程决策的纯规则模块

## 9. 会话记录增强

扩展 `InterviewRoundRecord`，至少保留以下字段：

- `analysisReason`
- `followUpDecision`
- `followUpDecisionReason`
- `missingPointsSnapshot`

本阶段这些字段主要供后端调试、回归和后续报告扩展使用，不要求前端展示。

## 10. 错误处理

原则：策略增强不能让主面试链路更脆弱。

- 分析层失败 -> 回退到当前基础启发式
- 决策层失败 -> 默认 `NEXT_QUESTION`
- LLM 文案生成失败 -> 使用固定兜底文案

约束：

- `/api/interviews/{sessionId}/answer` 不能因为追问策略升级而返回 500
- 最差结果只能是“策略效果退化”，不能是“面试流程中断”

## 11. 测试策略

### 11.1 `InterviewAnswerAnalyzer` 单测

覆盖：

- 缺关键点
- 深度不足
- 回答完整
- 前后矛盾
- 跑题
- 自我介绍类弱判定

### 11.2 `FollowUpDecisionEngine` 单测

覆盖：

- 普通题最多 1 次追问
- 高价值题最多 2 次追问
- 收尾阶段降低追问
- 明显错误优先澄清
- 已追问后仍低质量时跳题

### 11.3 `SimpleInterviewEngine` 集成测试

验证：

- `FOLLOW_UP / NEXT_QUESTION / END_INTERVIEW` 分支正确
- 轮次记录写入正确
- 追问方向与原因字段保留正确

### 11.4 回归测试

确保以下现有能力不回归：

- 创建会话
- 文本答题
- 生成报告
- 简历定制题目
- provider runtime

## 12. 验收标准

第一阶段验收通过需要满足：

- 同一输入下，追问决策稳定，不依赖 LLM 随机性
- 普通题不会连续无意义深挖
- 关键点明显缺失时会触发追问
- 完整回答时可以自然进入下一题
- 收尾阶段追问明显减少
- 主链路 `/answer` 可继续稳定工作
- 后端相关测试与 smoke/check 通过

## 13. 推荐实施顺序

建议分 3 个实现任务：

1. 先重构分析模型，补齐证据结构和单测
2. 再引入追问决策器，并接到 `SimpleInterviewEngine`
3. 最后补充轮次记录字段和集成回归测试

这样可以保证每一步都能独立验证，降低回归风险。
